User-Defined Ordinal Types

  • Ordinal Type: type in which the range of possible values can be easily associated with the set of positive integers.
    • Enumerations: type where all possible values are provided in the definition as named constants
      • Example in C#:      days{mon, tue, wed, thu, fri, sat, sun};
      • Design Issues:
        • Is an enum constant allowed to appear in more than one type def.
          • If so, how is the type of an occurrence of that constant checked?
        • Are enum values coerced to integer?
          • If so, there is little control over its range of values..
        • Any other type coerced to an enum type?
          • If an int were coerced to an enum type, an enum type could be assigned any integer value, whether it represented an enum constant or not
    • Subrange Type: ordered, contiguous subsequence of an ordinal type (supported in Ada)
      • Type compatability checked at compile time
      • Range checking done at run-time
      • Example in Ada:    
type Days is (mon, tue, wed, thu, fri, sat, sun);
subtype Weekdays is Days range mon..fri;
subtype Index is Integer range 1..100;
Day1: Days;
Day2: Weekdays;
Day2 := Day1;
  • Evaluation of Enumerated and Subrange Types:
    • Aids readability - Makes it clear that variables of enums or subrange can only store certain ranges of values
    • Aids reliability (compiler can check)
      • limits type operations to what is meaningful
      • Enums cannot be assigned outside its defined range (either the enum type or a subrange of it)
      • Ada, C#, Java 5 are better than C++ (Not coerced into integer types)
